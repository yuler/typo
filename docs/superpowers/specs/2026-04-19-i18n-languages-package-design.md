# i18n — `@typo/languages` Package & App Integration

**Date:** 2026-04-19
**Status:** Design approved, ready for implementation plan
**Scope:** Introduce a shared internationalization package and wire `apps/www` and `apps/desktop` to consume it. English, Simplified Chinese, and Japanese supported on day one.

---

## 1. Context & Scope

### 1.1 Current state

- Monorepo skeleton already in place (see `2026-04-17-monorepo-skeleton-design.md`): `pnpm-workspace.yaml` lists `apps/*`; no `packages/*` exists yet.
- `apps/desktop` — Tauri 2 + Vue 3 + Vite. All UI strings hardcoded in English across `windows/{Main,Settings,Upgrade}.vue` and shared components.
- `apps/www` — Astro 5 (static) with React islands and Tailwind v4. All page text hardcoded in English. `BaseLayout.astro` hardcodes `<html lang="en">`. `Header.astro` hardcodes `'Docs'`, `'Blog'`.
- No i18n infrastructure anywhere in the repo.
- `README.md` and `README.zh.md` exist (the only place a Chinese locale is currently named — uses `zh`).

### 1.2 This spec's scope

Create one shared package, `@typo/languages`, that owns:

- The translation strings for every app, organized by namespace.
- A small framework-agnostic runtime helper (`t`, `createTranslator`).
- A typed-key codegen step that fails CI if any locale is missing keys.

Then integrate it into both apps:

- `apps/www` — Astro built-in i18n routing (`/`, `/zh/`, `/ja/`), language switcher in the header, `<html lang>` and `<link rel="alternate" hreflang>` driven by current locale.
- `apps/desktop` — Vue composable wrapping the helper, locale persisted via the existing Tauri store, cross-window sync via Tauri events.

### 1.3 Decisions baked in (from brainstorm)

| Decision | Choice |
| --- | --- |
| Package role | Catalog **+** thin shared helper (typed keys, framework-agnostic `t`). No framework adapters in the package. |
| String organization | Namespaced by app + a shared `common` namespace. One file per `(namespace, locale)`. |
| Locale codes | Short codes: `en`, `zh`, `ja`. (Region-tagged variants like `zh-TW` deferrable as siblings.) |
| File format | JSON source-of-truth + generated `.d.ts` for typed keys. |
| Helper API | `t(locale, namespace, key, vars?)` plus `createTranslator(locale, namespace)`. `{var}` interpolation. Fallback to `en` + dev warn. |
| `apps/www` integration | Astro built-in i18n with URL-prefixed routing (`/`, `/zh/`, `/ja/`); SSG-rendered per locale. |
| `apps/desktop` integration | Custom Vue composable wrapping the helper. No `vue-i18n` dependency. |

Notes:

- Ships **raw TypeScript** via `exports` — both Vite (desktop) and Astro (www) already process workspace TypeScript natively. No `tsup`/`tsc` build artifact step.
- `prepare` runs on `pnpm install`, so a fresh clone always has `src/generated/keys.d.ts` even before any other script runs.
- `--experimental-strip-types` (Node 22+) avoids needing `tsx`. If `.nvmrc` later pins a version that doesn't support it, swap to `tsx` as a devDep.
- `private: true`, version pinned to monorepo (`1.2.0`), bumped via `bump.sh` (Section 7.4).

### 3.2 Public exports (`src/index.ts`)

```ts
export { t, createTranslator } from './t'
export {
  defaultLocale,
  localeNames,
  locales,
  type Locale,
} from './types'
export type { MessageKey, Namespace } from './generated/keys'
```

Per-namespace bundle exports (`@typo/languages/messages/<namespace>`) re-export the locale → messages map for that namespace. They exist for hypothetical future consumers that want raw message data without going through `t()` (e.g., a script generating a sitemap, an SSG that needs to enumerate every translated page title). Note: callers using `t()` import `lookup` (Section 3.7) which pulls in *all* namespace bundles, so per-namespace tree-shaking does not apply to the common path. Total payload across all three namespaces × three locales is well under 10 KB minified, so this is acceptable.

### 3.3 Locales (`src/types.ts`)

```ts
export const defaultLocale: Locale = 'en'

export const localeNames: Record<Locale, string> = {
  en: 'English',
```

### 3.4 Translation files

**Format:** flat-key JSON. One file per `(namespace, locale)`. No nesting.

**Rationale:** flat keys make every value a `string`, every diff a one-line change, and rule out the "leaf in one locale, object in another" foot-gun.

```json
// src/locales/common/en.json
{
  "brand.name": "typo",
  "action.save": "Save",
  "action.cancel": "Cancel",
  "language.en": "English",
  "language.zh": "中文",
  "language.ja": "日本語"
}
```

```json
// src/locales/common/zh.json
{
  "brand.name": "typo",
  "action.save": "保存",
  "action.cancel": "取消",
  "language.en": "English",
  "language.zh": "中文",
  "language.ja": "日本語"
}
```

Variable placeholders are written inline as `{name}`:

```json
{ "greeting": "Hello, {name}" }
```

**Initial namespace inventory** (will be filled in during implementation; here as scope guard):

- `common` — brand name, generic actions (save/cancel/ok/close), language display names. ~10 keys.
- `desktop` — settings labels, window titles emitted by Vue, ribbon strings, slash-command UI, upgrade dialog. ~30–50 keys.
- `www` — header nav, hero, feature grid copy, testimonials chrome, footer, language switcher labels. ~30–50 keys. **Long-form MDX bodies in `pages/blog/`, `pages/docs/` are explicitly out of scope** (Section 8).


(`desktop.ts` and `www.ts` are structurally identical, swap the namespace folder.)

### 3.6 The `t` function (`src/t.ts`)

```ts
import type { MessageKey, Namespace } from './generated/keys'
import type { Locale } from './types'
import { interpolate } from './interpolate'
import { lookup } from './lookup'

export function t<N extends Namespace>(
  locale: Locale,
  namespace: N,
  key: MessageKey<N>,
  vars?: Record<string, string | number>,
): string {
  const raw = lookup(locale, namespace, key)
  return vars ? interpolate(raw, vars) : raw
}

export function createTranslator<N extends Namespace>(locale: Locale, namespace: N) {
  return (key: MessageKey<N>, vars?: Record<string, string | number>): string =>
    t(locale, namespace, key, vars)
}
```

Namespace-as-explicit-arg keeps the typed-key generic per-namespace (so autocomplete on `MessageKey<'desktop'>` is a focused union, not a union of every key in every namespace), and forces call sites to be honest about which message bundle they depend on.

### 3.7 `lookup` — fallback policy (`src/lookup.ts`)

> **Note on namespace registration:** the `all` map below registers each namespace by name. Adding a new namespace later requires three edits: a new `<namespace>/{en,zh,ja}.json` triplet, a new `src/messages/<namespace>.ts` bundle, and one new line in this `all` map. The codegen automatically picks up new namespace folders for type generation; only the runtime registry below needs the manual touch.



```ts
import type { Namespace } from './generated/keys'
import type { Locale } from './types'
import { messages as common } from './messages/common'
import { messages as desktop } from './messages/desktop'
import { messages as www } from './messages/www'

const all: Record<Namespace, Record<Locale, Record<string, string>>> = {
  common, desktop, www,
}

export function lookup(locale: Locale, namespace: Namespace, key: string): string {
  const primary = all[namespace][locale]?.[key]
  if (primary !== undefined) return primary

  const fallback = all[namespace].en[key]
  if (fallback !== undefined) {
    if (typeof import.meta !== 'undefined' && import.meta.env?.DEV) {
      // eslint-disable-next-line no-console
      console.warn(`[@typo/languages] Missing "${key}" in ${namespace}/${locale}, using en fallback`)
    }
    return fallback
  }

  if (typeof import.meta !== 'undefined' && import.meta.env?.DEV) {
    // eslint-disable-next-line no-console
    console.warn(`[@typo/languages] Missing key "${key}" in ${namespace}`)
  }
  return key
}
```

The codegen `--verify` step (Section 3.9) means missing-key in non-en locales should never reach runtime — `lookup`'s fallback is the belt-and-braces safety net, not the primary defense.

### 3.8 `interpolate` (`src/interpolate.ts`)

```ts
const TOKEN = /\{(\w+)\}/g

export function interpolate(template: string, vars: Record<string, string | number>): string {
  return template.replace(TOKEN, (_, key) => {
    const value = vars[key]
    return value === undefined ? `{${key}}` : String(value)
  })
}
```

- Single brace, alphanumeric+underscore variable names.
- Missing var leaves the `{name}` placeholder intact (visible in dev, doesn't crash).

### 3.9 Codegen (`scripts/generate-types.ts`)

Reads each `<namespace>/en.json`. Emits `src/generated/keys.d.ts`:

```ts
export interface MessageKeys {
  common: 'brand.name' | 'action.save' | 'action.cancel' | 'language.en' | 'language.zh' | 'language.ja'
  desktop: /* union of desktop/en.json keys */
  www: /* union of www/en.json keys */
}

export type Namespace = keyof MessageKeys
export type MessageKey<N extends Namespace = Namespace> = MessageKeys[N]
```

Then it verifies:

- Every non-`en` locale file in each namespace contains the **same set of keys** as `en`.
- Every value is a non-empty string.
- Every `{var}` placeholder present in an `en` value is also present in every other locale's value for the same key.

On failure: print one error per offense with the offending file path and key, exit 1.

The script supports two modes:

- Default (no flag): regenerate `src/generated/keys.d.ts` and run all verifications.
- `--verify`: run all verifications **without** writing the output file. Used by CI to assert the committed file is in sync.

The generated file is **committed** to the repo. Trade-off: a regenerable artifact in git in exchange for zero install-time surprises and no codegen-runs-before-typecheck dance for app builds.

## 4. `apps/www` Integration

### 4.1 `astro.config.mjs` — i18n routing

```js
export default defineConfig({
  site: 'https://typo.yuler.cc',
  output: 'static',
  i18n: {
    defaultLocale: 'en',
    locales: ['en', 'zh', 'ja'],
    routing: {
      prefixDefaultLocale: false,
    },
    fallback: { zh: 'en', ja: 'en' },
  },
  integrations: [react(), mdx()],
  vite: {
    plugins: [tailwindcss()],
    resolve: {
      alias: {
        '@': fileURLToPath(new URL('./src', import.meta.url)),
      },
    },
  },
})
```

Astro's `fallback` means a page that exists only in `en` is reachable from a localized URL (it serves the en version), so we can roll out translations page-by-page.

### 4.2 Page structure

```
apps/www/src/pages/
├── index.astro                   # /          (en) — thin wrapper around <HomePage />
├── blog/
│   ├── index.astro               # /blog/      (en) — wrapper around <BlogIndex />
│   └── [...slug].astro           # /blog/<slug>
├── docs/
│   ├── index.astro               # /docs/      (en) — wrapper around <DocsIndex />
│   └── [...slug].astro           # /docs/<slug>

    ├── blog/
    │   ├── index.astro
    │   └── [...slug].astro
    └── docs/
        ├── index.astro
        └── [...slug].astro
```

**Required refactor — extract shared page components.** The current `apps/www/src/pages/index.astro` (and equivalents under `blog/` and `docs/`) inline the page composition (`<Header />`, `<Hero />`, etc.). To avoid duplicating that composition in `pages/[lang]/index.astro`, the implementation extracts each page's body into a sibling component under `apps/www/src/components/pages/` (e.g. `HomePage.astro`, `BlogIndex.astro`, `DocsIndex.astro`). Each `pages/*.astro` file (both root and `[lang]/`) becomes a one-line wrapper:

```astro
---
// apps/www/src/pages/index.astro (en, root)
import HomePage from '@/components/pages/HomePage.astro'
---
<HomePage />
```

```astro
---
// apps/www/src/pages/[lang]/index.astro (zh, ja)
import HomePage from '@/components/pages/HomePage.astro'
import { locales } from '@typo/languages'

export function getStaticPaths() {
  return locales.filter(l => l !== 'en').map(lang => ({ params: { lang } }))
}
---
<HomePage />
```

Inside each shared page component, strings come from `t()` via the `tr(Astro, 'www')` helper (Section 4.3), so the same component renders correctly for any locale; `Astro.currentLocale` is automatically populated by Astro's i18n routing. **For MDX pages under `blog/[...slug].astro` and `docs/[...slug].astro`, the chrome (header/footer/breadcrumbs) is localized but the MDX body content stays English in this spec** (see Section 8).

### 4.3 Locale resolution helper (`apps/www/src/lib/i18n.ts`)

```ts
import type { Locale, Namespace } from '@typo/languages'
import { createTranslator, defaultLocale } from '@typo/languages'

export function getLocale(astro: { currentLocale?: string }): Locale {
  return (astro.currentLocale as Locale) ?? defaultLocale
}

export function tr<N extends Namespace>(astro: { currentLocale?: string }, ns: N) {
  return createTranslator(getLocale(astro), ns)
}
```

`Astro.currentLocale` is populated by Astro's i18n routing — no manual URL parsing.

### 4.5 `BaseLayout.astro` — `<html lang>` + `hreflang`

```astro

import { locales } from '@typo/languages'
import '../styles/global.css'
```

The `<title>` and `description` props passed by pages should themselves come from `t()` at the page level so they're already localized when handed to the layout.

### 4.6 `LanguageSwitcher.astro`

A pure-Astro component (no React island) that emits anchor links rewriting only the locale segment of the current URL. Uses `Astro.url.pathname` and `Astro.currentLocale`. Lives next to `ThemeToggle` in `Header.astro`. Uses `localeNames` from `@typo/languages` for display labels.

### 4.7 What `apps/www` does *not* get

- **No JS-based language detection on first visit.** Visitor lands on `/` (en) and clicks the switcher to change languages. Site is `output: 'static'` — there's no server to act on `Accept-Language`.
- **No `localStorage` "remember language"** — the URL *is* the persistence layer.

## 5. `apps/desktop` Integration

### 5.1 Persisted `locale` field

Extend `DEFAULT_STORE` in `apps/desktop/src/store.ts`:

```ts
import type { Locale } from '@typo/languages'
import { defaultLocale } from '@typo/languages'

const DEFAULT_STORE = {
  // … existing fields
  locale: defaultLocale satisfies Locale,
}
```

Existing `initializeStore()` only writes a default if the key is absent — current installs migrate seamlessly on next launch.

### 5.2 `useI18n` composable (`apps/desktop/src/composables/useI18n.ts`)

```ts
import type { Locale, Namespace } from '@typo/languages'
import { emit, listen } from '@tauri-apps/api/event'
import { computed, ref } from 'vue'
import { createTranslator, defaultLocale } from '@typo/languages'
import { get, save, set } from '@/store'

const LOCALE_EVENT = 'typo://locale-changed'

const locale = ref<Locale>(defaultLocale)

export async function initializeI18n(): Promise<void> {
  locale.value = await get('locale')
  await listen<Locale>(LOCALE_EVENT, e => { locale.value = e.payload })
}

export async function setLocale(next: Locale): Promise<void> {
  locale.value = next
  await set('locale', next)
  await save()
  await emit(LOCALE_EVENT, next)
}

export function useI18n<N extends Namespace>(namespace: N) {
  const t = computed(() => createTranslator(locale.value, namespace))
  return { locale, t, setLocale }
}
```

Module-scoped `ref` = single source of truth across all components in the same window. Tauri `emit` + `listen` propagates the change to other windows (each window is its own webview process).

### 5.3 Wire-up in `App.vue`

```ts
import { initializeI18n } from '@/composables/useI18n'

onMounted(async () => {
  // … existing pre-store setup
  await initializeStore()
  await initializeI18n()
  initializeWindow()
  // … rest unchanged
})
```

```vue
    <option v-for="l in locales" :key="l" :value="l">{{ localeNames[l] }}</option>
  </select>
</template>
```

### 5.6 What `apps/desktop` does *not* get

- **No initial locale auto-detection from system locale.** New installs default to `en`; users opt in via Settings. (Could be added later as a one-call seed in `initializeStore` from `navigator.language`; keeping it out avoids confusing existing users on first re-launch.)
- **No translation of Tauri menu / system tray / window titles.** Those are read by Rust at startup from `tauri.conf.json` and aren't reachable from the Vue layer. Documented as a known limitation; addressing it requires passing the locale to Rust and reading the JSON catalog there — separate workstream.

## 8. Non-Goals

Explicitly out of scope for this spec, deferred to future cycles:

- **No translation of long-form MDX blog/docs content.** UI chrome only. Per-locale MDX with content-collection filtering is a separate workstream.
- **No translation of Tauri menu, system tray, window titles** (read by Rust at startup from `tauri.conf.json`).
- **No automatic locale detection** (browser `Accept-Language`, OS locale). New visitors land on English; users opt in.
- **No plural / select message format**. No `@formatjs/intl-messageformat`. Add when a real plural string appears.
- **No date / number / currency formatting helpers.** Each app uses native `Intl.*` APIs locally.
- **No `vue-i18n`.** Direct composable per design decision.
- **No translation management platform** (Crowdin / Lokalise / Weblate). JSON files edited in-repo via PRs.
- **No lazy loading of locales.** All inlined; total size is trivial for three locales.
- **No RTL support.** No RTL languages in the initial set.
- **No region-tagged variants.** `en`, `zh`, `ja` only. Adding `zh-TW` later requires a new locale value, new JSON files, no shape change.
- **No translation of `README.md`.** Separate workstream; `README.zh.md` already exists.
- **No edge redirect on `Accept-Language` for the marketing site.** Site is `output: 'static'`.
- **No upgrades to existing dependencies.** Strictly additive change.
- **No new runtime dependencies in either app.** `@typo/languages` is the only new dep, and it itself has no runtime deps.
- **No `tsconfig.base.json` extraction.** The package gets its own self-contained `tsconfig.json`. Extract a base when a third TS project needs the same options.
